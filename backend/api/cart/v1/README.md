# 🛒 购物车微服务核心接口设计规范与因果分析

在分布式、多端同步的电商微服务架构中，接口的设计不能仅满足于前端眼前的“页面展示”，更需要守住**分布式事务一致性**、**全盘营销计算**、**网络性能（RPC 开销）**以及**多端持久化**的底线。

---

## 1. 为什么购物车中必须有 `selected` (勾选状态) 字段？

### 典型请求模型

```protobuf
message AddProductToCartRequest {
  uint64 spu_id = 1;
  uint64 sku_id = 2;
  string merchant_id = 3;
  uint32 quantity = 4;
  bool selected = 5; // 核心：状态字段
}
```

### 🧠 原因分析与设计考量

- **多端状态持久化同步**：

  勾选状态属于**需要被持久化在数据库里的“业务状态”**，而不是前端内存的临时变量。用户在手机 App 勾选了 3 件商品（准备购买），由于手机没电换到 PC 网页端登录时，后端能够根据数据库中的 `selected` 状态完美复现用户的购买意向，避免用户重新在一堆历史商品中寻找并重新勾选，确保核心订单转化率。

- **后端“全盘营销价”计算的刚性需求**：

  购物车底部的“总金额、已减免金额”必须由后端的营销价格引擎（Promotion Service）实时计算，前端计算的金额永远不可信。后端需要通过 `selected = true` 的商品组合，来跨限界上下文计算**跨店满减（每满300减40）**、**优惠券可用性评估**以及**加价购触发条件**。如果数据库不存 `selected`，后端价格引擎就无法感知用户的“意向购买组合”。

- **结算页（CheckOut）的防刷新与轻量化数据对齐**：

  用户点击“去结算”跳转到订单确认页时，如果中途刷新页面或修改收货地址，结算页需要重新拉取商品列表。若后端存储了 `selected` 状态，结算页只需轻量调用 `GetSelectedCartItems` 即可。前端无需在每次跳转、刷新、改地址的 RPC 请求里，大包传递那一长串被勾选的 SKU 列表。

## 2. 为什么“添加商品到购物车”不能返回空结构体？

### 典型返回模型

```protobuf
message AddProductToCartResponse {
  uint32 cart_total_quantity = 1; // 购物车当前商品总数量
  uint64 cart_item_id = 2;        // 该商品在购物车中的唯一记录 ID
}
```

### 🧠 原因分析与设计考量

- **购物车“合并逻辑”与“总数限制”的实时校准**：

  购物车的容量不是无限的（例如上限 200 件）。当发生网络抖动、多端同时操作或触发上限时，后端必须通过错误码或返回值进行控制。

  此外，前端右上角购物车图标的“小红点数字”，绝不能由前端在本地盲目做 `+1` 操作（容易因为后端自动合并失效商品、满减自动加入赠品而导致数据不一致）。后端随接口返回最新的 `cart_total_quantity`，可以保障前端小红点**绝对的数据准确性**。

- **分布式环境下的“操作凭证（卡位）”与减少 RPC 浪费**：

  商品加入购物车成功后，在底层的 `cart_items` 表里已经诞生了一条带唯一主键的记录。后端必须将这个 **`cart_item_id`** 返回给前端。

  如果用户紧接着在页面上点击“数量 +1”或“删除”，前端可以直接拿着这个 `cart_item_id` 去调后端的更新/删除接口。如果不返回，前端在执行后续动作时，就必须被动地调用一次全量拉取购物车的接口（`GetCart`）来获取那条记录的 ID，造成了严重的网络与服务资源浪费。

## 3. 为什么“删除购物车商品”传 `cart_item_id` 比传 `sku_id` 更优雅？删除接口需要返回什么？

### 典型请求与返回模型

```protobuf
message RemoveFromCartRequest {
  uint64 cart_item_id = 1; // 推荐：使用购物车记录唯一 ID，而非 sku_id
}

message RemoveFromCartResponse {
  uint32 cart_total_quantity = 1; // 删除后，购物车剩下的商品总件数
  bool is_cart_empty = 2;         // 购物车是否已经彻底空了
}
```

### 🧠 原因分析与设计考量

- **天然支持“一键批量删除”与高性能 SQL 编写**：

  在购物车页面，用户高频进行的操作是“勾选多件商品 $\rightarrow$ 点击底部批量删除”。如果使用 `cart_item_id`，接口只需支持数组（`repeated uint64 cart_item_ids`），后端一条 `DELETE FROM cart_items WHERE id IN (...)` 即可搞定，性能极高。如果使用 `sku_id`，前端需要组装一个由 `sku_id` 和 `merchant_id` 组成的复杂结构体数组，后端的 SQL 会变得极其臃肿和低效。

- **避免同 SKU 多策略加入时的业务语义冲突**：

  在成熟电商中，同一个 `sku_id` 可能会分两行存在于同一个用户的购物车里（例如：一行是以正常价加入的商品，另一行是以换购价或促销套餐价加入的商品）。如果删除接口只传 `sku_id`，后端将无法分辨用户到底想删哪一行；而通过 `cart_item_id` 则能实现精确定位。

- **消除“盲人摸象”式的前端重复拉取开销**：

  删除接口返回 `cart_total_quantity`（最新总数）和 `is_cart_empty`（是否为空）。当用户在购物车删掉最后一件商品时，前端拿到返回值后，能够**当场、主动地**将页面切换为“购物车空空如也，去逛逛吧”的占位图，并刷新小红点。这就免去了前端再发起一次 `GetCart` 请求来判断购物车是否为空的额外网络开销。


---

## 总结

在进行电商微服务设计时，要坚守一个原则：**“接口返回值的每一次慷慨，都是为了避免下一次更沉重的全量网络拉取。”**

购物车系统的核心是**用轻量的增量状态（ID、Count、Status）去驱动前端的高频刷新**，从而将高并发下昂贵的全量拉取（`GetCart`）频率降到最低。
