## 购物车表设计

## 为什么选择这样的设计？

### 决策1：独立主键 + 业务唯一键，而不是只用 `(user_id, sku_id)` 作主键

- **方案A（采用）**：`id BIGSERIAL PRIMARY KEY` + `UNIQUE (user_id, merchant_id, sku_id)`

- **方案B（不采用）**：直接用 `(user_id, sku_id)` 作复合主键


**选择A的理由**：

- 自增主键让后续扩展（如关联优惠券、赠品等）更简单，外键引用更清晰。

- 业务唯一键保证了用户不会重复添加同一商品，重复添加时只需更新数量和时间。

- 复合主键不利于分布式扩展和分库分表。


### 决策2：使用商品快照字段，而不是只存 SKU ID 再去关联查询

- **方案A（采用）**：冗余 `spu_name`、`sku_name`、`price`、`thumbnail_url` 等快照字段

- **方案B（不采用）**：只存 `sku_id`，展示时实时查询商品服务


**选择A的理由**：

- **性能**：购物车列表是高频读操作，实时关联商品服务会增加延迟和耦合。快照数据让购物车服务可以独立响应，商品信息变更不影响已加入购物车的数据。

- **稳定性**：商品服务宕机时，购物车仍然可以展示（只是价格可能不是最新的，但下单前会再次校验）。

- **下单一致性**：下单时从购物车快照中获取价格等信息，与创建订单时的快照逻辑一脉相承。

- **代价**：更新价格时（如商品降价），购物车中的快照不会自动更新，需要额外处理（如提示用户“价格已变动”）。但这个代价远小于实时查询带来的复杂性和性能损耗。


### 决策3：`selected` 字段独立存在，而不是在下单时传递 SKU 列表

- **方案A（采用）**：购物车记录维护 `selected` 状态

- **方案B（不采用）**：下单时从前端传入选中的商品 ID 列表


**选择A的理由**：

- **跨端同步**：用户在 App 端勾选的商品，在网页端也能看到相同的选中状态。

- **减少网络传输**：下单时只需告诉后端“用我已选中的商品下单”，无需传递大量 SKU ID。

- **用户体验**：刷新页面后勾选状态不丢失。


### 决策4：`status` 字段管理生命周期，而不是硬删除

- **方案A（采用）**：使用 `active/expired/deleted` 状态标记

- **方案B（不采用）**：直接物理删除记录


**选择A的理由**：

- 可以分析用户的购物行为（哪些商品被删除了，哪些过期了），为推荐算法提供数据。

- 商品重新上架时可以恢复提醒。

- 物理删除不利于数据恢复和审计。


### 决策5：使用 `UUID` 作为 `user_id` 和 `merchant_id` 的类型

- **方案A（采用）**：`UUID` 类型，与 Casdoor 和订单表一致

- **方案B（不采用）**：`uint64` 自增 ID


**选择A的理由**：

- 保持整个系统中用户标识的一致性（订单、库存、购物车都是 `UUID`），避免类型转换和歧义。

- Casdoor 等身份认证系统通常使用字符串格式的用户 ID，UUID 是标准选择。


---

## 设计决策背后的取舍

|取舍|方案A（采用）|方案B（未采用）|理由|
|---|---|---|---|
|**冗余快照 vs 实时查询**|冗余快照|实时查询|用少量冗余数据换取高性能和稳定性|
|**独立主键 vs 复合主键**|独立主键+业务唯一键|复合主键|扩展性更好，后续拆分更容易|
|**`selected` 字段 vs 前端传参**|维护 `selected`|前端传参|跨端同步，体验更好|
|**逻辑删除 vs 物理删除**|状态标记|物理删除|数据可追溯，支持行为分析|
|**UUID vs 自增ID**|UUID|自增ID|与认证系统和订单表一致|

## 在微服务中的位置

购物车服务是一个独立的限界上下文，有自己的聚合根 `Cart`（包含一组 `CartItem`）。它与商品服务、库存服务的关系是：

- **商品服务**：在添加购物车时获取商品快照（通过同步调用或读取本地缓存），商品变更不直接影响购物车。

- **库存服务**：下单时由订单服务调用库存服务预占，购物车本身不直接与库存交互。

- **订单服务**：下单时读取购物车中的选中项，创建订单组和子订单，完成后清空已下单项。


这样的设计让购物车保持单一职责：记录用户想要购买的商品及其快照，管理选中状态，不参与库存扣减和支付逻辑。
